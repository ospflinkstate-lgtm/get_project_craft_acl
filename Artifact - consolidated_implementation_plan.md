# 📋 Consolidated Master Implementation Plan
> **โครงการ: ปรับปรุงประสิทธิภาพความมั่นคงปลอดภัย Firewall PoC & ระบบจัดเก็บแผนการติดตั้งอัตโนมัติ**
> เอกสารฉบับรวบรวม (Consolidated Master) อธิบายสถาปัตยกรรมเครือข่าย ปัญหาที่ได้รับการแก้ไข และแนวทางปฏิบัติงานทั้งหมด

## 🎯 1. ที่มาและวัตถุประสงค์ (Executive Summary)
โครงการนี้เกิดขึ้นเพื่อทดสอบระบบการบังคับใช้กฎ Firewall บนอุปกรณ์ **MikroTik CCR2116** ในโหมด **L2 Transparent Bridge (Firewall PoC)** เพื่อคัดกรองทราฟฟิกเข้าสู่เซิร์ฟเวอร์ในวง VLAN 21 (10.27.101.X) โดยมีวัตถุประสงค์ร่วมกับการวิเคราะห์และแก้ไขปัญหาขัดข้องทางเครือข่าย และการสร้างเครื่องมือเพื่อรายงานความคืบหน้าแก่ผู้บริหารและทีมวิศวกรแบบอัตโนมัติ

## 🛰️ 2. แผนงานระยะที่ 1: การวิเคราะห์ข้อมูลทราฟฟิกและการคัดกรองโฮสต์
วิเคราะห์ข้อมูล NetFlow ปริมาณมหาศาลจาก **OpenObserve** เพื่อนำมาแยกแยะความเสี่ยงของเซิร์ฟเวอร์จำนวน 83 เครื่องในวง VLAN 21:

*   **โหมดปลอดภัย (Bypassed / Safe Mode - 81 เซิร์ฟเวอร์):**
    *   เซิร์ฟเวอร์ทั่วไปที่ไม่มีส่วนเกี่ยวข้องกับ PoC จะถูกเปิดช่องทางผ่าน (Bypass) โดยใส่รายชื่อเข้าสู่ Address-List พิเศษ เพื่อให้อุปกรณ์ตรวจเช็กทราฟฟิกได้รวดเร็วและไม่สร้างผลกระทบต่อผู้ใช้งานปกติ
*   **เป้าหมายทดสอบ (PoC Targets - 2 เซิร์ฟเวอร์หลัก):**
    *   ทำการควบคุมและป้องกันอย่างเข้มงวดที่ **Safeline WAF (`10.27.101.41`)** และ **Backend Apache (`10.27.101.165`)**
*   **Zero-Trust Policy (กฎบล็อกข้อ `[99]`):**
    *   สร้างกฎฟายวอลล์ข้อสุดท้ายเพื่อทำการบล็อก (Drop) ทราฟฟิกที่เป็นความเสี่ยงทั้งหมดที่วิ่งมุ่งเป้ามายังเซิร์ฟเวอร์ PoC เพื่อความปลอดภัยสูงสุด

## 🛡️ 3. แผนงานระยะที่ 2: การวิเคราะห์และแก้ไขปัญหาเครือข่าย (Root Cause Solved)

จากการทดสอบระบบและวางระบบเครือข่ายในช่วงที่ผ่านมา ได้มีการแก้ไขข้อผิดพลาดสำคัญ 2 ประการดังนี้:

### 3.1 ปัญหา Firewall Counters ไม่ยอมนับและทราฟฟิกวิ่งทะลุบริดจ์
*   **สาเหตุ:** พอร์ตของ Bridge ใน MikroTik ถูกเปิดใช้งาน Hardware Offloading (`hw=yes`) ทำให้สวิตชิ่งชิปประมวลผลข้อมูลส่งต่อในเลเยอร์ 2 โดยไม่ผ่านขึ้นมาที่ CPU ส่งผลให้กฎไฟร์วอลล์ตรวจไม่พบแพ็กเก็ต
*   **แนวทางแก้ไข:**
    1. ทำการปิด Hardware Offloading ใน Bridge Ports (`hw=no`)
    2. เปิดฟีเจอร์ใช้งานไฟร์วอลล์บนบริดจ์ในระบบการตั้งค่า:
       ```routeros
       /interface bridge settings set use-ip-firewall=yes use-ip-firewall-for-vlan=yes
       ```

### 3.2 ปัญหา MikroTik ซิงก์เวลากับ NTP Server ไม่สำเร็จ (VLAN 201)
*   **สาเหตุ:** ตรวจพบการเปิดใช้งาน Extended ACL ชื่อ `LimitMgmtSW` บนอินเทอร์เฟซ VLAN 201 ของ Cisco Catalyst 9300 Core Switch ซึ่งเป็นด่านตรวจที่ไม่มีกฎอนุญาตทราฟฟิกพอร์ต UDP 123 (NTP) ทำให้แพ็กเก็ตถูกบล็อก (Implicit Deny)
*   **แนวทางแก้ไข:** ทำการแอดกฎเพิ่มบน Cisco Core Switch เพื่ออนุญาตให้ไอพีวงจัดการสามารถคุยกับ NTP Server ได้:
    ```config
    ip access-list extended LimitMgmtSW
     permit udp 10.27.201.0 0.0.0.255 host 10.27.1.201 eq 123
     permit udp host 10.27.1.201 eq 123 10.27.201.0 0.0.0.255
     permit udp 10.27.201.0 0.0.0.255 host 202.28.138.4 eq 123
     permit udp host 202.28.138.4 eq 123 10.27.201.0 0.0.0.255
    ```
