# 🤝 Work Concept & Collaboration Guidelines (with Antigravity)
> **เอกสารข้อตกลงและมาตรฐานการทำงานร่วมกับ AI (Antigravity)**
> *ไฟล์นี้เป็นเสมือน "กฎเหล็ก" สำหรับการทำงานในโฟลเดอร์/โปรเจกต์นี้ เพื่อให้สไตล์การทำงานเป็นไปในทิศทางเดียวกัน ปลอดภัย และตรวจสอบได้เสมอ*

---

## 📁 1. กฎการจัดการไฟล์และการสำรองข้อมูล (File Management & Versioning)
1. **Never Overwrite Blindly (ห้ามทับไฟล์เดิมทันที):** 
   - ก่อนที่ AI จะดัดแปลงโค้ดชุดใหญ่หรือสร้างไฟล์สำคัญทับของเดิม จะต้องทำการ Rename ไฟล์เดิมเพื่อแบ็คอัปไว้เป็นเวอร์ชันก่อนหน้าเสมอ (เช่น เปลี่ยน `script.py` เป็น `script_v1_backup.py`)
2. **Clear Naming Convention:** 
   - ตั้งชื่อไฟล์ให้สื่อความหมายชัดเจน หลีกเลี่ยงการใช้ชื่อแบบ `test.py` หรือ `new_file.html` ให้ใช้ชื่อที่บอกหน้าที่ของไฟล์แทน เช่น `extract_netflow_data.py`
3. **Workspace Cleanliness:**
   - ไฟล์ขยะหรือไฟล์ทดสอบ (Scratch) เมื่อใช้งานเสร็จและได้ผลลัพธ์ที่พอใจแล้ว ควรลบออกหรือย้ายไปโฟลเดอร์ Archive เพื่อไม่ให้โปรเจกต์รก

---

## 🌳 2. กฎการใช้งาน Git และ Version Control
1. **Manual Git Operations (รอดำเนินการ Git ภายหลัง):**
   - AI ยังไม่ต้องรันคำสั่ง `git add` และ `git commit` โดยพลการ ผู้ใช้จะเป็นคนสั่งการทำงานในส่วนของ Git ด้วยตัวเองอีกครั้งเมื่อพร้อม
2. **Descriptive Commit Messages:**
   - การ Commit ทุกครั้งต้องมีข้อความบอกเล่าที่ชัดเจน ว่า "เพิ่มอะไร", "แก้อะไร", หรือ "ลบอะไร" (เช่น `git commit -m "feat: เพิ่มระบบค้นหาตาม Port ใน acl_builder.html"`)
3. **Strict `.gitignore` Policy:**
   - ไฟล์ Data ขนาดใหญ่ (เช่น `.json`, `.csv` ที่มีขนาดหลักร้อย MB) หรือไฟล์ที่มีความอ่อนไหว **ห้ามนำเข้า Git เด็ดขาด** AI ต้องตรวจสอบ `.gitignore` เสมอก่อนดำเนินการ

---

## 🛡️ 3. กฎความปลอดภัย (Security First)
1. **No Hardcoded Tokens:**
   - ห้าม AI ฝังรหัสผ่าน, API Key, หรือ Git Token ลงในซอร์สโค้ด (Source Code) เด็ดขาด หากต้องใช้ ให้แนะแนวทางการอ่านจากไฟล์แยก หรือให้ผมกรอกเอง
2. **Double Check Destructive Actions:**
   - ทุกครั้งที่มีคำสั่งที่เกี่ยวกับการ **ลบข้อมูล (Delete/Remove)** หรือการเขียนทับระบบสำคัญ AI ต้องเตือนให้เห็นชัดเจนและรอผมกดยืนยันก่อนเสมอ

---

## 💬 4. รูปแบบการสื่อสารและการทำงาน (Communication Rules)
1. **Propose Before Act (เสนอแผนก่อนลงมือทำ):**
   - หากงานมีความซับซ้อน AI ต้องสร้าง `implementation_plan.md` เพื่อสรุปแนวคิดเป็นข้อๆ ให้ผมเห็นภาพรวมและอนุมัติก่อนเริ่มเขียนโค้ด
2. **Bite-Sized Execution (ทำทีละขั้นตอน):**
   - ไม่เขียนโค้ดยาวรวดเดียวจบ (ถ้าไม่จำเป็น) ให้ทำทีละเฟส ทดสอบให้ผ่าน แล้วค่อยขยับไปเฟสถัดไป เพื่อให้ง่ายต่อการหาบั๊ก (Troubleshooting)
3. **Clear & Professional Explanation:**
   - การอธิบายโค้ดหรือการทำงาน ต้องใช้ภาษาที่เข้าใจง่าย มีการยกตัวอย่าง มีตารางเปรียบเทียบ หรือใช้สัญลักษณ์ (Emojis) ช่วยจัดระเบียบข้อมูลให้ดูเป็นมืออาชีพ
4. **Ask if Unsure (ถามเมื่อไม่แน่ใจ):**
   - หากมีข้อมูลที่ไม่ชัวร์ หรือไม่แน่ใจในความต้องการ ให้กลับมาถามผมก่อนลงมือทำ หรือก่อนเขียนโค้ดเสมอ ห้ามเดาเอาเอง
5. **Context Awareness (อ้างอิงข้อมูลเดิม):**
   - AI ต้องพยายามอ้างอิงรูปแบบการทำงานจาก Knowledge Base หรือโค้ดเดิมในโปรเจกต์ (เช่น ไฟล์ `Skill-NW&SE.md` และไฟล์ `Concept_Progress_Implement_Detail.md`) ก่อนที่จะเสนอแนวทางใหม่ เพื่อให้เกิดความต่อเนื่อง

---
